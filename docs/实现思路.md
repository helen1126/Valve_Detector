## 1. 问题定义与建模思路

### 1.1 回归而非分类

阀门开度是一个**连续物理量**，将 0°~80° 离散为 N 个类别会引入两个问题：

1. **粒度矛盾**：分类类别数越多，每个类别的样本越少；类别数越少，精度上限越低（如分为 8 类，最好情况也只能达到 ±5°）
2. **丢失有序性**：分类损失（交叉熵）不区分"预测 10° 真实 20°"和"预测 70° 真实 20°"的严重程度差异

因此本项目将角度检测建模为**回归任务**，输出连续值，直接优化 MAE，目标 MAE ≤ 1°。

### 1.2 Sigmoid 输出与角度归一化

模型最后一层使用 `Sigmoid`，输出值域 [0, 1]。与之对应，标签角度通过线性归一化映射到 [0, 1]：

```
归一化：   y = (angle - angle_min) / (angle_max - angle_min)
反归一化： angle = y * (angle_max - angle_min) + angle_min
```

这种设计有三个好处：

- **天然约束**：Sigmoid 输出永远在 [0, 1]，不会预测出负角度或超范围角度
- **数值稳定**：Sigmoid + MSE/L1 的组合在反向传播时数值稳定，避免梯度爆炸
- **范围可配**：通过 `angle_min`/`angle_max` 配置即可适配不同角度范围（如 0°~90°、0°~360°），无需改模型结构

### 1.3 0°~80° 范围的设定

工业阀门实际工作范围通常在 0°~80° 之间（完全关闭 0°，全开约 80°），超过 80° 的场景极少。将范围设为 0°~80° 而非 0°~90° 或 0°~360°，可以在 Sigmoid 输出分辨率不变的情况下，提升有效角度区间的精度。

### 2.3 数据流闭环

**训练流**：
```
原始图像（编号_角度.jpg）
  → 文件名解析得到角度标签
  → 视角推断（top/side）
  → Side 过采样（all_view 时）
  → 双 transform 增强（side 用更强增强）
  → 角度归一化到 [0,1]
  → 模型前向 → Sigmoid 输出 [0,1]
  → 组合损失（L1+MSE）× Focal 加权 × 视角加权
  → 反向传播 → AdamW + CosineAnnealingLR
  → (可选) Stage 2：side_view 微调
  → 保存 checkpoint
```

**预测流**：
```
输入图片/视频
  → 模型加载（自动检测架构）
  → (可选) 智能裁剪 / 多尺度推理
  → 预处理 → 模型推理 → Sigmoid
  → 反归一化得到角度
  → 绘制标注图
  → 输出结果
```

**部署流**：
```
API 请求
  → 模型热加载检查
  → 图片解码
  → 调用 ValvePredictor.predict_single
  → 标注图 base64 编码
  → JSON 响应
```

## 3. 模型设计

### 3.1 工厂模式 + 注册表

`models/__init__.py` 使用注册表模式管理 8 种模型：

```python
MODEL_REGISTRY = {
    "resnet50": build_resnet,
    "resnet101": build_resnet,
    "efficientnet_b4": build_efficientnet,
    "efficientnet_b5": build_efficientnet,
    "convnext_base": build_convnext,
    "convnext_large": build_convnext,
    "swin_base": build_swin,
    "swin_large": build_swin,
}

def build_model(model_name, pretrained=True, dropout=0.2, freeze_backbone=False):
    builder = MODEL_REGISTRY[model_name]
    return builder(model_name, pretrained, dropout, freeze_backbone)
```

**设计动机**：

- **统一接口**：`build_model(model_name)` 一行代码构建任意模型，训练/预测/评估脚本无需关心具体实现
- **易于扩展**：新增模型只需实现 `build_xxx` 函数并注册，无需修改调用方
- **ONNX 导出统一**：`export_onnx()` 函数对所有模型通用

### 3.2 统一回归头设计

所有模型共享相同的回归头结构模式：

```
骨干网络（移除分类头，输出特征）
  → [LayerNorm 或 Dropout]
  → Linear(features → 512)
  → [GELU 或 ReLU]
  → Dropout(0.2)
  → Linear(512 → 1)
  → Sigmoid
```

**设计动机**：

- **特征提取与回归解耦**：骨干网络负责特征提取，回归头负责将特征映射到角度，职责清晰
- **512 维瓶颈层**：相比直接 `features → 1`，增加一个 512 维中间层提升表达能力，同时通过 Dropout 防止过拟合
- **Sigmoid 约束输出范围**：与角度归一化对应，保证预测值合法

### 3.3 Transformer 与 CNN 回归头的差异

不同骨干网络的回归头在归一化层和激活函数上有细微差异，遵循各自架构的最佳实践：

| 骨干类型 | 归一化层 | 激活函数 | 权重初始化 | 理由 |
|----------|----------|----------|------------|------|
| ConvNeXt / Swin | LayerNorm | GELU | trunc_normal_(std=0.02) | Transformer 系列标配，与骨干一致 |
| ResNet / EfficientNet | Dropout | ReLU | kaiming_normal_ | CNN 系列标配，与骨干一致 |

**设计动机**：回归头的风格与骨干保持一致，避免归一化/激活不匹配导致的训练不稳定。

### 3.4 ConvNeXt 默认推荐的选型理由

项目默认使用 `convnext_base`，理由：

1. **工业视觉表现优异**：ConvNeXt 在 ImageNet 等基准上超越同参数量的 ResNet/EfficientNet
2. **现代架构**：LayerNorm + GELU + Depthwise Conv，训练稳定
3. **预训练质量高**：使用 `fb_in22k_ft_in1k`（22k 类别预训练 + 1k 微调），特征表达能力强
4. **推理效率**：相比 Swin（固定窗口大小），ConvNeXt 对不同输入尺寸更灵活

## 4. 数据策略

### 4.1 文件名解析约定

数据集文件命名格式：`编号_角度.后缀`（如 `0001_6.7.jpg` 表示 6.7°）。

**设计动机**：

- **零配置标注**：无需额外的标注文件（CSV/JSON），标注信息直接编码在文件名中
- **防错**：文件名与图片一一对应，不会出现标注错位
- **可读性**：人工查看文件名即可知道角度，便于排查问题

正则解析：`(\d+)_(\d+\.?\d*)\.` 提取编号和角度。

### 4.2 视角推断与 Side 过采样

**视角推断**：从文件路径中检测 `side_view` / `top_view` 关键字判断视角。

**Side 过采样**：当训练 `all_view`（混合视角）时，side 样本通常远少于 top 样本。过采样逻辑：

```python
repeat = ceil(n_top / n_side) - 1
samples = samples + side_samples * repeat
```

**设计动机**：

- Side 视角样本量少，模型容易偏向 top 视角
- 单纯复制样本会导致过拟合，但配合 Side 专用增强（见 4.3）可以产生足够的多样性
- 相比 class_weight 加权，过采样 + 增强的效果更好（加权只是放大梯度，过采样让模型多见不同增强后的样本）

### 4.3 双 transform 机制

为 side 视角构建独立的增强流水线（`train_transforms_side`），相比通用增强：

- `Perspective`（透视变形）概率翻倍，模拟 side 视角的透视失真
- `OpticalDistortion`（光学畸变）概率翻倍，模拟 side 视角的镜头畸变

**设计动机**：Side 视角的物理特性（透视、畸变、遮挡）与 top 视角差异巨大，统一的增强策略无法同时适配两种视角。双 transform 让 side 样本见到更多符合其物理特性的增强样本。

### 4.4 不使用旋转/翻转的领域知识

数据增强流水线**刻意排除**旋转（`Rotate`）和翻转（`Flip`）。

**设计动机**：

- 阀门角度是**有方向的物理量**：旋转图片 90° 后，原本 30° 的阀门看起来像 120°，但标签仍是 30°
- 旋转/翻转会破坏角度标签的语义正确性，导致模型学到错误映射
- 这与一般图像分类任务不同（分类任务中旋转/翻转是常用增强）

### 4.5 训练时 ValveSmartCrop

`data/augmentation.py` 中的 `ValveSmartCrop` 以 p=0.3 的概率在训练时裁剪放大阀门区域。

**设计动机**：

- 推理时智能裁剪会改变图像分布（从"远景"变为"近景"）
- 如果训练时模型从未见过裁剪后的图像，推理时智能裁剪反而会降低精度（训练-推理分布不一致）
- 训练时以一定概率应用相同的裁剪，让模型见过两种分布，推理时无论是否裁剪都能稳定预测

## 5. 训练策略

### 5.1 组合损失 0.7×L1 + 0.3×MSE

```python
per_sample_loss = 0.7 * mae + 0.3 * mse
```

**设计动机**：

- **L1（MAE）**：对误差线性惩罚，梯度恒定，训练稳定，但对小误差不敏感
- **L2（MSE）**：对大误差平方惩罚，能压制异常预测，但对小误差梯度太小
- **组合**：L1 为主保证整体稳定性，L2 为辅压制大误差样本
- **权重 0.7:0.3**：L1 占主导（直接优化目标 MAE），L2 辅助

关键实现细节：`reduction='none'` 保留逐样本损失，是后续视角加权和 Focal 加权的前提。

### 5.2 视角加权

```python
sample_weights = [5.0 if view == "side" else 1.0 for view in views]
per_sample_loss = per_sample_loss * sample_weights
```

**设计动机**：

- Side 视角学习难度远高于 top（透视、畸变、遮挡），且样本量少
- 不加权时，模型会"放弃" side 视角（损失占比小，梯度被 top 主导）
- 5.0 倍权重让 side 样本的损失贡献与 top 相当，强迫模型重视 side 视角

### 5.3 Focal-style 难样本挖掘

```python
if focal_gamma > 0 and training:
    abs_error = mae.detach()  # 停止梯度
    focal_weight = abs_error ** gamma
    focal_weight = focal_weight / focal_weight.mean()  # 归一化
    per_sample_loss = per_sample_loss * focal_weight
```

**设计动机**：

- 即使视角加权后，仍有一些"特别难"的样本（如极端畸变、严重遮挡）
- Focal 加权让误差越大的样本权重越高，自动聚焦难样本
- `gamma=2.0`：误差 2° 的样本权重是误差 1° 的 4 倍

**关键细节**：

- `detach()`：权重不参与反向传播，否则会导致梯度计算错误（权重依赖误差，误差依赖权重，形成循环）
- `mean()` 归一化：保证总损失量级不变，避免加权后学习率需要重新调
- 仅训练时启用：验证时使用等权损失，反映真实性能

### 5.4 两阶段训练

```
Stage 1: all_view 数据正常训练 → 保存最佳 checkpoint
Stage 2: 加载 Stage 1 最佳模型 → side_view 数据 → lr=1e-5 微调 30 epochs
```

**设计动机**：

- Stage 1 让模型学到通用的角度检测能力（top + side）
- Stage 2 专门针对 side 视角微调，低学习率避免破坏 Stage 1 学到的 top 视角能力
- 相比单纯加权，两阶段训练更激进地优化 side 视角，同时通过低学习率保护 top 视角

### 5.5 渐进式解冻

```python
if warmup_epochs > 0:
    # 前 N 个 epoch：冻结骨干，只训练回归头
    if current_epoch < warmup_epochs:
        freeze_backbone()
    else:
        unfreeze_backbone()
        lr = lr * 0.1  # 解冻后降低学习率
```

**设计动机**：

- 预训练骨干的特征提取能力已经很强，初期不需要大幅调整
- 冻结骨干时只训练回归头，显存占用小、训练快，适合快速验证
- 解冻后降低学习率，避免大幅梯度破坏预训练特征

### 5.6 自适应 batch_size

```python
def auto_batch_size(model_name, image_size, gpu_memory_mb):
    vram_per_sample = MODEL_VRAM_PER_SAMPLE[model_name]
    scaled_vram = vram_per_sample * (image_size ** 2) / (384 ** 2)
    available = gpu_memory_mb - 1500  # 预留 1.5GB
    batch_size = int(available / scaled_vram)
    return max(2, min(128, batch_size // 8 * 8))  # 对齐到 8 的倍数
```

**设计动机**：

- 不同模型、不同图像尺寸的显存占用差异巨大，手动指定 batch_size 容易 OOM 或浪费显存
- 自适应计算让用户无需关心显存细节，`--batch_size 0` 即可自动选择最优值
- 对齐到 8 的倍数：GPU 张量核心对 8 的倍数有加速

## 6. 推理优化

### 6.1 智能裁剪三级检测策略

`utils/image_optimization.py` 的 `detect_valve_region` 采用三级 fallback 策略：

| 优先级 | 策略 | 原理 | 适用场景 |
|--------|------|------|----------|
| 1 | 颜色特征检测 | HSV 空间提取绿/红掩码 | 阀门有明确颜色特征（最可靠） |
| 2 | 显著性检测 | 频谱残差显著性 | 阀门颜色不典型但与背景对比明显 |
| 3 | 边缘密度检测 | 8×8 网格 Canny 边缘密度 | 阀门结构复杂、边缘丰富 |

**设计动机**：

- 没有单一策略能覆盖所有场景（不同工厂的阀门颜色、背景差异大）
- 三级 fallback 保证鲁棒性：颜色最可靠，失败时退到显著性，再失败退到边缘密度
- 显著性检测优先尝试 `opencv-contrib` 的 `StaticSaliencySpectralResidual`，不可用时回退到自实现的 `_simple_saliency`（纯标准 cv2），避免强制依赖 opencv-contrib

### 6.2 多尺度自适应加权融合

```python
valve_ratio = bbox_area / image_area
crop_weight = min(0.9, max(0.5, 1.0 - valve_ratio * 3.3))
fused = pred_original * (1 - crop_weight) + pred_cropped * crop_weight
```

**设计动机**：

- **远距离场景**（阀门占比小，如 2%）：原图中阀门太小，模型主要看到背景；裁剪图放大后阀门清晰，应更信任裁剪图 → `crop_weight = 0.9`
- **近距离场景**（阀门占比大，如 15%）：原图已经足够清晰，裁剪可能丢失上下文；应更信任原图 → `crop_weight = 0.5`
- **自适应**：根据阀门占比动态调整权重，无需用户手动选择

### 6.3 视频抽帧推理

支持两种抽帧方式：

- `fps`：每秒抽 N 帧（如 `fps=2` 每秒 2 帧）
- `frame_interval`：每 N 帧抽 1 帧（如 `frame_interval=30` 每 30 帧 1 帧）

**设计动机**：

- 视频帧率差异大（15fps ~ 60fps），`fps` 模式按时间抽帧，适合需要固定时间间隔的场景
- `frame_interval` 模式按帧抽帧，适合需要固定帧间隔的场景（如动作分析）
- 两种模式覆盖不同需求，默认每秒 1 帧（平衡精度与处理速度）

### 6.4 模型热加载与架构自动检测

**模型热加载**（`check_and_reload`）：
- 每次预测请求检查模型文件 mtime
- 文件更新时自动重新加载，无需重启 API

**架构自动检测**（`_load_pytorch_model`）：
- 从 checkpoint 的 `hyper_parameters.model_name` 读取模型架构
- 用户无需手动指定 `--model_name`，避免架构不匹配导致的维度错误

**设计动机**：

- 生产环境中模型迭代频繁，热加载实现"无停机更新"
- 架构自动检测消除人为错误（训练用 convnext_large，预测时忘记指定导致维度不匹配）

## 7. 服务化部署

### 7.1 API 接口职责划分

| 接口 | 职责 | 设计考量 |
|------|------|----------|
| `POST /predict` | 单张预测 | 核心接口，支持 smart_crop/multi_scale 参数覆盖 |
| `POST /predict/batch` | 批量预测 | 复用单张逻辑，返回逐张结果 + 总耗时 |
| `POST /predict/video` | 视频抽帧 | 保存临时文件后调用 predict_video，避免内存爆炸 |
| `GET /health` | 健康检查 | 轻量级，不加载模型也能响应 |
| `GET /info` | 模型信息 | 返回模型配置，便于前端适配 |

**设计动机**：

- 单张/批量/视频分离：不同场景的请求模式不同（视频是长耗时请求），分离便于后续做差异化限流
- 健康检查不依赖模型：即使模型加载失败，健康检查也能响应，便于监控系统判断服务存活

### 7.2 HTTPS 双服务模式

配置 `SSL_KEYFILE` + `SSL_CERTFILE` 后，`asyncio.gather` 同时启动：

- **HTTPS:8443** — 完整 API 服务（带 SSL）
- **HTTP:8000** — 完整 API 服务（不带 SSL，**不重定向**）

**设计动机**：

- **避免 Cloudflare 隧道重定向循环**：Cloudflare 隧道指向 HTTP 端口时，如果 HTTP 端口做 308 重定向到 HTTPS，会形成"浏览器→Cloudflare→HTTP(308)→浏览器→Cloudflare→HTTP(308)..."死循环
- **双服务不重定向**：HTTP 端口直接提供完整 API，Cloudflare 隧道走 HTTP 即可，浏览器侧的 HTTPS 由 Cloudflare 边缘提供
- **HTTPS 端口独立可用**：不经过 Cloudflare 的场景（如内网直连）可以直接用 HTTPS

### 7.3 前端直接复用 ValvePredictor

Streamlit 前端**不调用 API**，而是直接 `import ValvePredictor` 在进程内推理。

**设计动机**：

- **本地 Demo 场景**：前端与后端在同一台机器，走 HTTP 会增加无谓的网络开销
- **简化部署**：本地 Demo 只需启动 Streamlit，无需同时启动 API
- **生产场景分离**：生产环境中前端（浏览器）与后端（服务器）分离，此时走 API；两种场景各取所需

## 8. 技术栈选型

| 技术 | 选型理由 |
|------|----------|
| **PyTorch Lightning** | 标准化训练流程（训练/验证/测试循环、回调、日志），减少样板代码；支持混合精度、梯度裁剪、多 GPU |
| **Albumentations** | 最快的数据增强库（比 torchvision 快 3~5 倍）；支持自定义 Transform（ValveSmartCrop）；丰富的增强类型 |
| **timm** | 最全的预训练模型库（ConvNeXt/Swin/EfficientNet 等）；统一的 `create_model` 接口；高质量预训练权重 |
| **FastAPI** | 基于 Pydantic 的自动文档生成；原生异步支持；性能优于 Flask |
| **Streamlit** | 几行代码构建交互式 Demo；适合机器学习可视化；无需前端知识 |
| **ONNX** | 跨平台部署（Windows/Linux/嵌入式）；ONNX Runtime 推理加速 2~3 倍；支持 CUDA/CPU 自动切换 |
| **loguru** | 零配置日志（开箱即用）；支持文件轮转、颜色输出、异常堆栈完整捕获 |
| **OpenCV（headless）** | 图像处理标准库；headless 版本无 GUI 依赖，适合服务器部署 |
